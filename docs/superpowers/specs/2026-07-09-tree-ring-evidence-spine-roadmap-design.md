# Tree Ring Evidence Spine Roadmap Design

## Status

Approved brainstorming direction: build the next Tree Ring Memory roadmap
around an Evidence Spine. The priority order is:

1. Trust and proof first.
2. Operator cockpit second.
3. Adoption and integration third.

This spec is a roadmap and decomposition document for the next A/B/C lanes:

- TUI and integrations link cockpit.
- Harness certification matrix.
- Recall quality dashboard.

The roadmap is intentionally split into implementation phases. It should not
land as one large PR.

## Context

Tree Ring Memory now has a Rust-native CLI, SQLite store, TUI, local
certification script, integration scan, DOX/Revolve adapters, and shared action
request/report contracts for durable CLI/TUI operations. The current proof
surface is real but scattered:

- `scripts/certify-tree-ring.sh` writes certification summaries and metrics
  under `target/tree-ring-certification/`.
- `tree-ring integrations scan` detects project and home markers for common
  agent harnesses.
- The TUI can show integration scan results but does not yet browse proof
  artifacts or guide the operator through next steps.
- Recall performance is measured by certification, but recall quality and
  ranking trust are not yet a first-class operator surface.

The next roadmap should turn those pieces into a coherent proof workflow
without adding hidden background writers, daemon behavior, or harness-specific
core coupling.

## North Star

Tree Ring Memory should make proof inspectable inside the tool itself.

An operator should be able to open the TUI, inspect current evidence, see what
is missing or stale, run explicit confirmed refreshes, and understand whether
Tree Ring is healthy for the current project and target agent harnesses.

Proof is generated by bounded commands and scripts. The TUI surfaces that proof
and can request a refresh only through explicit confirmation.

## Goals

- Create a stable file-backed Evidence Spine that certification, harness
  probes, recall diagnostics, docs, and the TUI can share.
- Preserve existing `metrics.json` compatibility while adding richer evidence
  indexes beside it.
- Add a TUI evidence browser that reads existing evidence by default.
- Let the TUI initiate evidence refreshes only through explicit confirmed
  actions.
- Represent harness compatibility as evidence, not marketing claims.
- Represent recall quality as reviewable diagnostics, not only latency.
- Keep the roadmap decomposed into small implementation lanes.

## Non-Goals

- Do not add a daemon, background recorder, or hidden proof runner.
- Do not change the SQLite schema or public JSONL memory schema.
- Do not make the TUI run long commands silently.
- Do not make core memory logic depend on Codex, Claude Code, OpenCode, Goose,
  Pi, Agent Zero, or any other harness.
- Do not replace the current certification metrics format.
- Do not redesign the whole TUI layout in the first phase.
- Do not claim compatibility unless a local evidence producer generated a
  pass, fail, or skip record.

## Roadmap Architecture

The roadmap uses an Evidence Spine as the shared foundation.

```text
Evidence producers
  ├─ certification script
  ├─ harness probes
  └─ recall quality diagnostics
        ↓
file-backed evidence records
  ├─ metrics.json remains compatible
  ├─ evidence-index.json points at latest records
  ├─ harness/*.json records probe status
  └─ recall-quality/*.json records query/ranking diagnostics
        ↓
evidence readers
  ├─ TUI evidence browser
  ├─ CLI report commands, if needed later
  ├─ docs/status updates
  └─ CI or PR summaries
```

The Evidence Spine should be a small, typed reader/writer boundary rather than
a database. It can start in the CLI crate because the first consumers are the
certification script and TUI. If the model later needs wider reuse, it can move
to a dedicated crate without changing file shapes.

## Evidence Model

Evidence records should be structured, local, and easy to inspect.

### Certification Summary

Certification evidence records the local proof already produced by
`scripts/certify-tree-ring.sh`:

- overall status
- timestamp
- release binary size
- project install size
- global install size
- CLI import throughput
- 10k and 30k insert throughput
- 10k and 30k recall average and max latency
- enabled, skipped, and failed optional smokes
- paths to summary, metrics, and raw output files

The current `metrics.json` shape remains compatible. New evidence files should
reference it instead of replacing it.

### Harness Probe Result

Harness evidence records whether Tree Ring is usable from a target agent
surface. Each record includes:

- harness id such as `codex`, `claude-code`, `opencode`, `goose`, `pi`, or
  `agent-zero`
- detected markers from integration scan
- command or probe attempted
- status: `pass`, `fail`, or `skip`
- timestamp
- evidence path
- failure summary or skip reason
- next step for the operator

Harness probes must not modify third-party harness configuration by default.
They should prove that the current project has enough guidance and command
surface to use Tree Ring, then produce a clear next step if not.

### Recall Quality Result

Recall quality evidence records whether retrieval is useful, not just fast.
Each record includes:

- query set id
- query text
- expected memory ids, labels, or source refs
- returned memory ids
- rank positions
- score breakdown when available
- latency
- status: `pass`, `fail`, or `needs_review`
- notes for ambiguous or qualitative cases

The first recall-quality diagnostics should use curated local fixtures or
generated test stores so the results are deterministic enough to trust.

### Evidence Index

`evidence-index.json` points to the latest known evidence records by category.
The TUI reads this first so it can open quickly without scanning every file.

The index should include:

- generated timestamp
- latest certification record path
- latest harness record paths by harness id
- latest recall quality record path
- top-level status rollups
- stale or missing evidence flags

## TUI Workflow

The TUI adds an Evidence mode.

Primary flow:

1. `/evidence` opens the evidence browser.
2. The main list shows evidence categories and recent records:
   certification, harness probes, recall quality.
3. The detail pane shows the selected record: metrics, logs path, timestamp,
   status, and failure or skip notes.
4. A checklist area suggests next actions:
   refresh certification, run harness probes, run recall diagnostics, or open
   integration scan.
5. Any refresh action creates a `PendingAction` confirmation before running.

The TUI reads existing evidence by default. It should show a clear empty state
when no evidence index exists, including the command that would generate the
first certification evidence.

Refresh behavior should be explicit and bounded:

- If an action is safe to run inside the TUI, it must show visible status while
  running and write structured evidence when done.
- If an action is too long-running or risky for the first implementation, the
  TUI should present the exact external command instead of attempting to run it.

## Phase 1: Evidence Spine And TUI Evidence Browser

Phase 1 is the first implementable sub-project.

Scope:

- Add typed evidence index models and readers.
- Update certification to write an evidence index beside existing outputs.
- Add `/evidence` TUI mode.
- Render certification metrics, evidence paths, timestamps, and skip reasons.
- Add explicit pending action for refresh certification, implemented either as
  a bounded command runner or as an operator-facing next-step command if command
  execution risk is too high.

Acceptance criteria:

- Existing `metrics.json` consumers continue to work.
- TUI opens with existing evidence and no background work.
- Missing evidence has a useful empty state.
- Certification evidence includes install size and recall speed.
- Tests cover parsing/rendering evidence and the TUI mode transition.
- Certification still passes after the changes.

## Phase 2: Harness Certification Matrix

Phase 2 adds harness proof records.

Scope:

- Add harness probe evidence producers for Codex, Claude Code, OpenCode, Goose,
  Pi, and Agent Zero/A0.
- Use integration scan markers as probe inputs.
- Produce pass, fail, or skip records with evidence paths and next steps.
- Surface harness probe details in `/evidence`.
- Link failed or skipped probes to integration-scan next steps.

Acceptance criteria:

- Each harness has a deterministic local probe that does not mutate harness
  configuration by default.
- Probe output is structured and indexed.
- TUI shows probe status and evidence path per harness.
- Skip states are explicit and not treated as passes.
- Docs and PR summaries can cite evidence paths instead of compatibility
  claims.

## Phase 3: Recall Quality Dashboard

Phase 3 adds retrieval trust diagnostics.

Scope:

- Define curated recall quality query sets.
- Generate deterministic stores or fixtures for quality checks.
- Record returned ids, rank positions, score breakdown, and latency.
- Mark results as pass, fail, or needs review.
- Surface recall-quality evidence in `/evidence`.
- Later, optionally add a compact trust summary to the TUI home screen.

Acceptance criteria:

- Recall diagnostics prove both speed and relevance.
- Failures include enough detail to debug ranking or filtering behavior.
- Quality evidence does not require real private memories.
- The TUI can show ranking evidence without leaking sensitive payloads.

## Data Flow

Certification flow:

```text
scripts/certify-tree-ring.sh
  -> existing summary.md and metrics.json
  -> evidence-index.json certification entry
  -> TUI evidence reader
  -> evidence browser detail pane
```

Harness flow:

```text
integrations scan
  -> harness probe command
  -> harness evidence record
  -> evidence-index.json harness entry
  -> TUI evidence browser and checklist
```

Recall-quality flow:

```text
query fixture or generated store
  -> recall diagnostic runner
  -> recall-quality evidence record
  -> evidence-index.json recall-quality entry
  -> TUI evidence browser
```

## Error Handling

- Missing evidence index: show an empty state and first command to run.
- Malformed evidence file: show a parse error with the file path; do not crash
  the TUI.
- Stale evidence: show timestamp and stale marker, but keep the record
  inspectable.
- Skipped probe: show skip reason and next step; do not count it as a pass.
- Failed refresh: write or show the failure summary, leave previous evidence
  available, and avoid partial success claims.
- Sensitive recall diagnostics: redact or omit payload text unless explicitly
  safe fixture data is used.

## Testing Strategy

Phase 1 tests:

- Evidence index deserialize/serialize tests.
- Certification writer test that preserves `metrics.json` and writes the new
  index.
- TUI `/evidence` parser and mode transition tests.
- Render-buffer smoke for empty and populated evidence states.
- Refresh confirmation tests for any TUI-triggered action.

Phase 2 tests:

- Harness probe fixtures for detected, available, pass, fail, and skip states.
- Integration-scan linkage tests.
- TUI evidence rendering for per-harness records.

Phase 3 tests:

- Recall diagnostic fixtures with expected rank positions.
- Sensitive payload redaction tests.
- TUI evidence rendering for recall quality records.

Final verification for each phase should include:

- `cargo fmt --check`
- `cargo test --locked`
- `cargo clippy --locked --all-targets`
- `git diff --check`
- `sh scripts/certify-tree-ring.sh` when certification behavior changes

## Sequencing Decision

The roadmap is approved as Evidence Spine first. The first implementation plan
should cover Phase 1 only. Phase 2 and Phase 3 should each get their own
follow-up spec or implementation plan after Phase 1 is merged and verified.

## Scope Guard

This roadmap should not reopen shared action foundation work unless a Phase 1
implementation requires a small action wrapper for evidence refresh. It should
also not bundle harness probes or recall diagnostics into Phase 1. Phase 1
establishes the evidence model and TUI browser so later proof producers have a
stable place to report.
